01

The Problem

Wildfires are accelerating in both frequency and severity across the American West. In 2023 alone, over 2.7 million acres burned across California, Oregon, and Arizona — with fire seasons now extending year-round rather than peaking in summer months.

The critical bottleneck is detection latency. Traditional fire monitoring relies on satellite passes every 12–24 hours combined with spotty manual aerial coverage. By the time a fire is confirmed and response dispatched, containment windows have often already closed.

Commercial systems like Pano AI and ALERTCalifornia use persistent camera networks and ML classifiers to push detection under 15 minutes — but require expensive physical infrastructure. The research question: can a satellite-driven, cloud-native pipeline approach that speed using freely available federal data?

Project Goal

Build an end-to-end geospatial data pipeline ingesting NASA thermal anomaly data every 10 minutes, cross-validated against Sentinel-2 imagery and NOAA weather feeds, with a CNN smoke classifier flagging high-probability wildfire events — targeting detection latency under 30 minutes at zero infrastructure cost beyond cloud compute.

Each layer is an isolated Docker container. Airflow manages inter-layer dependencies and failure recovery.

The PostGIS database is the spatial backbone. All geometries — FIRMS thermal points, Sentinel-2 raster footprints, and NWS fire weather zones — are stored in WGS84 (EPSG:4326) with GIST spatial indexing for sub-second bounding box queries and cross-source spatial joins. Three core Airflow DAGs run the pipeline: firms_ingest every 10 minutes, sentinel_download triggered by new thermal events intersecting available imagery tiles, and fire_score running CNN inference on assembled multi-source event records.

03

Technical Implementation

Each pipeline stage is written in Python with strict separation of concerns. Below are representative extracts from the core ingestion and spatial cross-validation stages.

ingest/firms_collector.py Python 
# Fetch NASA FIRMS thermal anomaly data and load into PostGIS
import geopandas as gpd
from sqlalchemy   import create_engine
from shapely.geometry import Point

FIRMS_URL = "https://firms.modaps.eosdis.nasa.gov/api/country/csv/"
MAP_KEY   = "YOUR_FIRMS_MAP_KEY"

def fetch_firms_events(aoi: str, days: int = 1) -> gpd.GeoDataFrame:
    """Download VIIRS SNPP NRT thermal anomaly points for last N days."""
    url = f"{FIRMS_URL}{MAP_KEY}/VIIRS_SNPP_NRT/{aoi}/{days}"
    df  = gpd.read_file(url)
    df["geometry"] = df.apply(
        lambda r: Point(r["longitude"], r["latitude"]), axis=1
    )
    return gpd.GeoDataFrame(df, crs="EPSG:4326")

def load_to_postgis(gdf: gpd.GeoDataFrame, table: str):
    """Append new records to PostGIS — skips duplicates via primary key."""
    engine = create_engine("postgresql://user:pass@localhost/wildfire_db")
    gdf.to_postgis(
        table, engine, if_exists="append",
        index=False, chunksize=1000
    )
analysis/spatial_crossvalidation.sql PostGIS SQL 
-- Cross-validate FIRMS thermal events against Sentinel-2 tiles
-- and attach NWS fire weather context
SELECT
    f.acq_date,
    f.acq_time,
    f.frp                   AS fire_radiative_power,
    f.confidence,
    ST_AsText(f.geom)       AS location_wkt,
    s.tile_id               AS sentinel_tile,
    w.zone_name             AS fire_weather_zone,
    w.fire_danger_rating

FROM   firms_events    f
JOIN   sentinel_tiles  s  ON ST_Intersects(f.geom, s.footprint)
JOIN   nws_fire_zones  w  ON ST_Within(f.geom, w.geom)

WHERE  f.confidence        >= 70
  AND  f.acq_date          >= NOW() - '24 hours'::interval
  AND  w.fire_danger_rating IN ('HIGH', 'VERY HIGH', 'EXTREME')

ORDER BY f.frp DESC;

The CNN classifier is a fine-tuned EfficientNet-B3 trained on labeled Sentinel-2 scene patches. Each patch is a 256×256 crop across bands B2, B3, B4, B8, and B12 — chosen to capture smoke in visible, NIR, and SWIR. Events scoring above 0.65 are flagged as high-confidence wildfire candidates and written back to PostGIS for dashboard display.

05

Technical Challenges

⏱
Multi-Source Temporal Alignment
FIRMS refreshes every 10 minutes but Sentinel-2 revisit time is 5 days. Airflow DAG logic handles events with no matching imagery tile — falling back to NOAA-only scoring and flagging for human review rather than silently dropping events.
Solved: Airflow conditional branching
📡
CRS Consistency Across Sources
NASA FIRMS uses WGS84, Sentinel-2 tiles ship in UTM (zone varies by tile), and some NWS datasets use EPSG:5070 (Albers). All geometries must be normalized to EPSG:4326 before PostGIS insertion to prevent silent spatial join mismatches.
Solved: Rasterio CRS normalization layer
⛈
Cloud Cover Masking Sentinel-2 Imagery
Fire events under heavy smoke or cloud cover produce low-quality tiles that degrade CNN scores. Scene-level cloud cover percentage is now checked from Sentinel-2 metadata before downloading — scenes above 40% cloud cover are excluded and handled by FIRMS-only scoring.
Solved: Cloud cover metadata pre-filter
💰
GCP Egress Cost at Scale
Full Sentinel-2 tiles are 500–800 MB each. Downloading full tiles for every FIRMS event is cost-prohibitive. Solution: extract only a 256×256px bounding-box crop around the event centroid using COG range requests, reducing per-event download to under 2 MB.
Solved: COG range requests via Rasterio
